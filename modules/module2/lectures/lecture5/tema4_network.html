Security Zones в банківській архітектурі

Багаторівнева ізоляція критичних систем

Internet Zone (Untrusted)

Рівень довіри: Немає | Доступ: Публічний

  • Public websites, mobile apps
  • Максимальна exposure, мінімальна довіра

DMZ (Demilitarized Zone)

Рівень довіри: Низький | Доступ: Controlled

  • Web servers, API gateways, load balancers
  • Firewall з обох сторін (Internet і Internal)

Application Zone (Internal)

Рівень довіри: Середній | Доступ: Authenticated

  • Application servers, microservices, CBS core
  • Доступ тільки з DMZ через firewall

Data Zone (Highly Restricted)

Рівень довіри: Високий | Доступ: Strictly controlled

  • Database servers, data warehouses
  • Доступ тільки з Application Zone
  • Encryption at rest обов'язково

Management Zone (Admin)

Рівень довіри: Максимальний | Доступ: Admin only

  • Jump hosts, monitoring, backup systems
  • VPN + MFA + privileged access management

DMZ Design Patterns

Типова архітектура DMZ для банку

Three-Tier DMZ (найпопулярніша):

Internet → Edge Firewall (WAF, DDoS protection)
↓
DMZ: Load Balancers, API Gateway, Web Servers
↓
Internal Firewall (stateful inspection, IPS)
↓
Internal Network: CBS, Application Servers
↓
Database Firewall → Database Zone

Компоненти DMZ:

  • WAF (Web Application Firewall): F5, Imperva, Cloudflare - захист від OWASP Top 10
  • Load Balancers: NGINX, HAProxy, F5 BIG-IP - traffic distribution + SSL termination
  • API Gateway: Kong, Apigee - rate limiting, authentication, routing
  • IDS/IPS: Snort, Suricata, Palo Alto - threat detection
Golden Rule: DMZ ніколи не повинна мати direct access до database zone

Firewall Rules та Policies

Принцип least privilege

Базові принципи:

  • Default Deny: Блокувати все, дозволяти тільки необхідне
  • Least Privilege: Мінімальні необхідні permissions
  • Stateful Inspection: Tracking connection state
  • Application-Aware: Layer 7 inspection (не тільки ports)

Приклад Firewall Rules (Internet → DMZ):

Source Destination Port Action
Any DMZ Web Servers 443 (HTTPS) ALLOW
Any DMZ API Gateway 443 (HTTPS) ALLOW
Any DMZ 22, 3389, etc DENY
Any Any Any DENY (default)

Приклад Rules (DMZ → Internal):

Source Destination Port Action
DMZ Web Servers App Servers 8080 (HTTP) ALLOW
DMZ API Gateway CBS API 8443 (HTTPS) ALLOW
DMZ Database Servers Any DENY

Micro-segmentation

Granular security на рівні workload

Traditional Segmentation vs Micro-segmentation:

Traditional (Perimeter-based)

  • Network-level segmentation (VLANs, subnets)
  • Firewall між zones
  • East-West traffic не контролюється
  • Lateral movement легкий після breach

Problem: "Flat" internal network

Micro-segmentation (Zero Trust)

  • Workload-level policies
  • Every workload = isolated segment
  • East-West traffic контролюється
  • Lateral movement блокується

Solution: Zero Trust inside network

Micro-segmentation Technologies:

  • VMware NSX: Software-Defined Networking для micro-segmentation
  • Cisco ACI: Application Centric Infrastructure
  • Kubernetes Network Policies: Pod-to-pod segmentation
  • Illumio: Adaptive segmentation platform
  • Service Mesh (Istio, Linkerd): Micro-segmentation для microservices
⚠️ 80% breaches використовують lateral movement - micro-segmentation це блокує

Software-Defined Networking (SDN)

Програмована мережа для динамічної segmentation

SDN Architecture:

Application Layer: Network applications, security policies, analytics
↕️ Northbound API (RESTful)
Control Layer: SDN Controller (VMware NSX Manager, Cisco APIC)
↕️ Southbound API (OpenFlow, OVSDB)
Infrastructure Layer: Physical/Virtual switches, routers

Переваги SDN для банків:

  • Dynamic policies: Зміна правил без touching hardware
  • Automation: Policy as Code, CI/CD для network
  • Visibility: Повний traffic visibility
  • Agility: Швидке створення segments
  • Cost: Менше hardware, більше software
  • Security: Micro-segmentation на рівні VM/Container
Trend: 60% банків впроваджують SDN для micro-segmentation (Gartner 2024)

Zero Trust Network Access (ZTNA)

"Never trust, always verify"

Принципи Zero Trust:

1. Verify Explicitly

Автентифікація та авторизація на кожному запиті

  • MFA обов'язково
  • Device posture check
  • Continuous verification

2. Least Privilege

Мінімальний необхідний доступ

  • Just-in-time access
  • Just-enough access
  • Час-обмежені sessions

3. Assume Breach

Проектувати як би breach вже є

  • Micro-segmentation
  • Encryption everywhere
  • Continuous monitoring

ZTNA vs Traditional VPN:

Критерій Traditional VPN ZTNA
Trust model Network-based (inside = trusted) Identity-based (never trust)
Access Full network access Application-specific
Lateral movement Easy Blocked
Performance Backhauling через gateway Direct connection
✅ ZTNA знижує attack surface на 70-80%

ZTNA Implementation для CBS

Практичне впровадження Zero Trust

ZTNA Components:

Identity Provider (IdP)

  • Okta, Azure AD, Auth0
  • Centralized authentication
  • MFA enforcement
  • SSO (Single Sign-On)

ZTNA Gateway

  • Zscaler, Cloudflare Access, Palo Alto Prisma
  • Application connector
  • Policy enforcement point
  • Traffic inspection

Endpoint Agent

  • Device posture check
  • OS версія, antivirus status
  • Encryption status
  • Compliance verification

Policy Engine

  • Centralized policy management
  • Role-based access
  • Context-aware decisions
  • Continuous evaluation

Access Flow:

1. User → Login (MFA) → IdP

2. IdP → Issues identity token → User

3. User → Request CBS access → ZTNA Gateway

4. ZTNA Gateway → Check policy (identity, device, location, time) → Policy Engine

5. Policy Engine → ALLOW/DENY → ZTNA Gateway

6. ZTNA Gateway → Establish encrypted tunnel → CBS Application

7. Continuous monitoring та re-evaluation

Best Practice: Start with high-risk applications (CBS, payment systems) перед full rollout

Network Security Monitoring

Continuous visibility та threat detection

Monitoring Stack:

Flow Analysis:

  • NetFlow/sFlow: Traffic patterns
  • Tools: SolarWinds, Plixer Scrutinizer
  • Use case: Anomaly detection, bandwidth monitoring

Packet Capture:

  • Full packet capture: Wireshark, tcpdump
  • Storage: 7-30 днів retention
  • Use case: Forensics, troubleshooting

IDS/IPS:

  • Signature-based: Known attack patterns
  • Anomaly-based: ML для unusual behavior
  • Tools: Snort, Suricata, Palo Alto Threat Prevention

SIEM Integration:

  • Log aggregation: Firewall, IDS, servers
  • Correlation: Multi-source threat detection
  • Tools: Splunk, QRadar, Sentinel

Key Metrics to Monitor:

  • Firewall denies: Spike = potential scan/attack
  • Failed authentication attempts: Brute force detection
  • Unusual traffic patterns: Data exfiltration, C2 communication
  • Port scans: Reconnaissance activity
  • Geo-location anomalies: Access from unexpected countries
⚠️ Average detection time: 207 днів. Network monitoring скорочує до <24 годин

Network Security Compliance

Регуляторні вимоги для банків

PCI DSS Requirements:

  • Req 1: Install and maintain firewall configuration
  • Req 1.1: Network diagram документований та актуальний
  • Req 1.2: Firewall rules обмежують traffic між trusted/untrusted networks
  • Req 1.3: DMZ для public-facing systems
  • Req 1.4: Personal firewalls на mobile devices

ISO 27001 Controls:

  • A.13.1: Network security management
  • A.13.1.1: Network controls (firewalls, segmentation)
  • A.13.1.2: Security of network services
  • A.13.1.3: Segregation in networks (DMZ, zones)

NIST Cybersecurity Framework:

  • PR.AC-5: Network integrity захищений (segmentation, separation)
  • DE.CM-1: Network monitoring для anomalies
  • DE.AE-1: Network baseline встановлено для виявлення відхилень
Audit готовність: Network diagrams, firewall configs, change logs повинні бути актуальними
