Частина 1: Вибір об'єкта дослідження

Завдання 1.1: Обрати український банк для аналізу

Що потрібно зробити:

Оберіть один український банк для детального аналізу його IT-архітектури. При виборі враховуйте наступні фактори:

  • Доступність інформації: Банк повинен мати достатньо публічної технічної інформації (API документація, технічні статті, кейси впровадження)
  • Технологічна зрілість: Бажано обрати банк з сучасною архітектурою або цікавими технологічними рішеннями
  • Розмір та складність: Великі банки мають складнішу архітектуру, але більше інформації; цифрові банки простіші, але інноваційніші

Рекомендовані варіанти з обґрунтуванням:

  • ПриватБанк - найбільший банк України з потужною IT-інфраструктурою
    • Переваги: багато публічної інформації, відкрите API, численні технічні статті
    • Особливості: складна legacy-система + сучасні цифрові сервіси
  • Monobank - перший повністю мобільний банк України
    • Переваги: cloud-native архітектура, відкрите API, активна технічна спільнота
    • Особливості: мікросервісна архітектура, використання AWS
  • Ощадбанк - державний банк з масштабною інфраструктурою
    • Переваги: великі обсяги даних, складні інтеграції з державними системами
    • Особливості: поєднання legacy-систем з модернізацією
  • ПУМБ - універсальний банк з гібридним підходом
    • Переваги: баланс між традиційним та цифровим банкінгом
    • Особливості: поступова міграція на хмарні технології
  • А-Банк - інноваційний банк з фокусом на технології
    • Переваги: впровадження нових технологій, експерименти з AI
    • Особливості: активне використання аналітики та machine learning

Після вибору банку необхідно:

  1. Зібрати базову інформацію про банк (розмір, кількість клієнтів, основні продукти)
  2. Знайти технічні матеріали про банк (офіційний сайт, API документація, технічні блоги)
  3. Визначити ключові технологічні рішення банку (мобільний додаток, інтернет-банкінг, API)
  4. Скласти попередній список компонентів системи для подальшого аналізу
💡 Порада: Почніть з офіційного сайту банку, розділів для розробників, API документації. Також пошукайте статті на DOU, технічні презентації на SlideShare, відео з конференцій на YouTube.

Результат виконання: Обґрунтований вибір банку з переліком знайдених джерел інформації (мінімум 5 джерел).

Частина 2: Моделювання архітектури

Завдання 2.1: Створення високорівневої діаграми архітектури

Що потрібно зробити:

Створити візуальну модель архітектури обраного банку, яка відображає всі ключові компоненти та їх взаємозв'язки. Діаграма повинна бути інформативною, але не перевантаженою деталями.

Покрокова інструкція створення діаграми:

  1. Визначте основні шари архітектури:
    • Presentation Layer (канали доступу): веб-банкінг, мобільний додаток, банкомати, відділення
    • Channel Integration Layer: омніканальна платформа, API Gateway
    • Business Logic Layer: сервіси обробки транзакцій, бізнес-правила
    • Core Banking Layer: система управління рахунками, процесинг
    • Data Layer: бази даних, сховища даних, аналітичні системи
    • Integration Layer: ESB, middleware, зовнішні інтеграції
  2. Додайте ключові компоненти в кожен шар:
    • Frontend: React/Angular додатки, iOS/Android apps
    • Backend: мікросервіси або монолітні модулі
    • Middleware: черги повідомлень (Kafka, RabbitMQ)
    • Databases: SQL (Oracle, PostgreSQL), NoSQL (MongoDB, Cassandra)
    • External Systems: SWIFT, платіжні системи, кредитні бюро
  3. Позначте потоки даних та інтеграції:
    • Синхронні виклики (REST API, SOAP)
    • Асинхронні повідомлення (messaging queues)
    • Batch-процеси (нічні звірки, генерація звітів)
    • Real-time потоки (streaming data)
🛠️ Рекомендовані інструменти для створення діаграм:
  • draw.io - безкоштовний онлайн редактор
    • Використовуйте шаблони: Software → AWS Architecture або Network
    • Бібліотеки: UML, AWS, Azure для банківських компонентів
  • Mermaid Live Editor - діаграми як код
    • Простий синтаксис для швидкого прототипування
    • Приклад коду для початку роботи
  • PlantUML - для детальних UML діаграм
    • Component diagrams для архітектури
    • Deployment diagrams для інфраструктури

Частина 3: Аналіз безпеки

Завдання 3.1: Проведення Threat Modeling за методологією STRIDE

Що потрібно зробити:

Систематично проаналізувати архітектуру банку на предмет потенційних загроз безпеці, використовуючи методологію STRIDE від Microsoft. Необхідно ідентифікувати загрози для кожного компонента та запропонувати заходи протидії.

Методологія STRIDE - детальне пояснення:

  • S - Spoofing (Підробка ідентичності):
    • Загроза: Зловмисник видає себе за легітимного користувача або систему
    • Приклади: фішинг, підробка сесій, man-in-the-middle атаки
    • Що аналізувати: механізми автентифікації, управління сесіями, сертифікати
  • T - Tampering (Несанкціонована зміна даних):
    • Загроза: Модифікація даних під час передачі або зберігання
    • Приклади: SQL injection, зміна параметрів транзакцій, модифікація логів
    • Що аналізувати: цілісність даних, контрольні суми, цифрові підписи
  • R - Repudiation (Відмова від дій):
    • Загроза: Користувач заперечує виконання транзакції
    • Приклади: відмова від платежу, заперечення отримання коштів
    • Що аналізувати: системи логування, аудит trail, незаперечність транзакцій
  • I - Information Disclosure (Розкриття інформації):
    • Загроза: Несанкціонований доступ до конфіденційних даних
    • Приклади: витік даних клієнтів, розкриття паролів, крадіжка номерів карток
    • Що аналізувати: шифрування, контроль доступу, маскування даних
  • D - Denial of Service (Відмова в обслуговуванні):
    • Загроза: Унеможливлення доступу легітимних користувачів до сервісів
    • Приклади: DDoS атаки, вичерпання ресурсів, блокування акаунтів
    • Що аналізувати: масштабованість, rate limiting, резервування
  • E - Elevation of Privilege (Підвищення привілеїв):
    • Загроза: Отримання несанкціонованих прав доступу
    • Приклади: експлуатація вразливостей, privilege escalation, обхід RBAC
    • Що аналізувати: контроль доступу, принцип найменших привілеїв, сегментація

Покрокова інструкція аналізу:

  1. Крок 1: Декомпозиція системи
    • Розділіть систему на зони довіри (trust boundaries)
    • Ідентифікуйте точки входу (entry points)
    • Визначте потоки даних між компонентами
    • Класифікуйте дані за рівнем конфіденційності
  2. Крок 2: Ідентифікація загроз
    • Для кожного компонента пройдіться по всіх категоріях STRIDE
    • Запитайте: "Як зловмисник може реалізувати цю загрозу?"
    • Документуйте кожну виявлену загрозу
  3. Крок 3: Оцінка ризиків
    • Ймовірність: Низька / Середня / Висока
    • Вплив: Мінімальний / Помірний / Високий / Критичний
    • Ризик = Ймовірність × Вплив
  4. Крок 4: Розробка контрзаходів
    • Для кожної загрози визначте існуючі та рекомендовані контролі
    • Пріоритизуйте за співвідношенням ризик/вартість

Завдання 4.2: Створення матриці ризиків

Що потрібно зробити:

На основі виявлених вразливостей створити матрицю ризиків для візуалізації та пріоритизації загроз.

Методологія оцінки ризиків:

  1. Визначення ймовірності (Likelihood):
    • Дуже низька (1): Теоретично можливо, але малоймовірно (потребує insider access)
    • Низька (2): Можливо за певних умов (складна експлуатація)
    • Середня (3): Реалістично (відомі випадки в індустрії)
    • Висока (4): Ймовірно (активно експлуатується)
    • Дуже висока (5): Майже гарантовано (автоматизовані атаки)
  2. Оцінка впливу (Impact):
    • Мінімальний (1): Незначні незручності, без фінансових втрат
    • Низький (2): Обмежений вплив, втрати до $10K
    • Середній (3): Помітний вплив, втрати $10K-$100K
    • Високий (4): Серйозний вплив, втрати $100K-$1M
    • Критичний (5): Катастрофічний вплив, втрати >$1M або репутаційний крах
  3. Розрахунок ризику:
    • Risk Score = Likelihood × Impact
    • 1-5: Низький ризик (зелений)
    • 6-10: Середній ризик (жовтий)
    • 11-15: Високий ризик (помаранчевий)
    • 16-25: Критичний ризик (червоний)

Шаблон документування рекомендації:

REC-001: Впровадження багатофакторної автентифікації (MFA)

Пріоритет: P0 - Критичний (Must Have)

Проблема:
Виявлено відсутність MFA для критичних операцій в інтернет-банкінгу. Це створює високий ризик компрометації акаунтів через фішинг або крадіжку паролів.

Рішення:
Впровадити обов'язкову багатофакторну автентифікацію для:

  • Входу в систему інтернет-банкінгу
  • Підтвердження транзакцій понад 1000 грн
  • Зміни критичних налаштувань акаунту
  • Адміністративного доступу співробітників

Технічна реалізація:

  • Основний метод: TOTP (Google Authenticator, Microsoft Authenticator)
  • Резервний метод: SMS OTP (з попередженням про ризики)
  • Для VIP клієнтів: Hardware tokens (FIDO2)
  • Інтеграція: OAuth 2.0 + OIDC

Вартість впровадження:

  • Ліцензії: $50,000/рік (Auth0 або Okta)
  • Розробка: 320 людино-годин
  • Тестування: 80 людино-годин
  • Навчання: 40 людино-годин

ROI та вигоди:

  • Зниження fraud losses на 80% (~$500K/рік)
  • Відповідність PSD2 та НБУ вимогам
  • Підвищення довіри клієнтів
  • Зниження витрат на support (password reset)

KPI успіху:

  • 100% coverage для транзакцій >1000 грн
  • 90% adoption rate серед активних користувачів
  • <0.1% fraud rate після впровадження
  • <2% відмов через технічні проблеми

Ризики впровадження:

  • Опір користувачів (мітигація: поступове впровадження)
  • Технічні складності інтеграції (мітигація: PoC спочатку)
  • Втрата клієнтів (мітигація: educa campaign)
